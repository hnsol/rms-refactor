## 問題解決はRMSの土台

前の章で、マネジャーの役割は問題解決であると確認しました。ところで、問題解決とは何でしょうか。どこまでを問題と呼び、何をどう進めるのが「問題解決」でしょうか。

思い出してみてください。皆んなで、いい話をしているのに動きが止まることがあります。会議は成立しているのに、何も決まらない。悪い人がいるわけでもないのに、結果が出ない。そんな経験、ありますよね。

「やっている感」はあるけれども、なんだか「進んでいる」気はしない。論点がずれているのでしょうか？ 原因と症状を取り違えているのでしょうか？ 感覚派が多くて理論派が少ないからこうなってしまうのでしょうか？

結果が出ない理由ははっきりしています。「原因」に対して対策できていないのです。表面的な症状に手を打っても、肝心の原因に届かなければ変化は起きません。インフルエンザに罹ったときに、風邪薬を飲んでも治りません。原因はウィルスですから、抗インフルエンザ薬が効果的です。

RMSの土台には、停滞をほぐすための「問題解決の作法」があります。ここから先のすべて、RMSのスキルもテクニックも、問題解決が前提になっています。

ただし、RMSの問題解決は「ひとりで解く技」ではありません。部下・上司・他部署・他社など、関係者の混成チームで進めるものです。だからこそ、個人技だけで押し切らず、合意と実行までを扱います。

## 1. 問題解決とは「ありたい姿と現状の差を埋めること」

マネジメントにおける「問題」の定義は世界共通です。1978年にノーベル経済学賞を受賞したハーバート・A・サイモンは、問題を「ありたい姿（To Be）と現状（As Is）の差」と定義しました。問題解決は、その差を見つけ、埋めるための道筋を描く行為です。

現場でよく起きるのは、困りごと・症状・感情が混ざってしまうこと。だから最初に「何が問題か」を揃えるところから始めます。ここでのポイントは、「問題には定義がある」ことと「問題解決には王道手順がある」ことです。

いきなり解決策に飛びつくと、立場や目的がズレたまま議論が進み、結局は手戻りになります。結果も出ません。問題解決は、思いつきではなく設計なんです。

## 2. RMSでやることは「世界共通の手順」をチームで回す

問題解決の手順は世界共通です。問題→真因→対策の順番で進める。このシンプルな型を、混成チームでもスムーズに回せるようにするのがRMSです。

しかし、現場では順番が逆になりがちです。いきなり対策を出す、あるいは「誰が悪いか」に引き寄せられる。だからRMSでは、まず問題を定義し、次に真因を掘り、最後に対策を設計します。

RMSが目指すのは、個人のひらめきではなく、チームの合意と実行です。部下でも上司でも、他部署でも他社でも、同じイメージを抱いて進める。結果が出なければ、チームはうまく回りません。そのために必要な交通整理がRMSです。その状態をつくることが、RMSの役割なんですね。

## 3. 落とし穴：誤解1つ、失敗1つ

よくある誤解は「問題解決は正しい答え探し」だと思い込むこと。実際には、正解は1つではなく、合意可能で実行できる解をつくる営みです。答え合わせより、原因について、対策についての合意づくりが大事になります。

もう1つの失敗は「分析で満足して実行に移らない」こと。資料がきれいでも、動かなければ現実は変わりません。だから問題解決は、実行までの一連を設計する必要があります。

## 4. 参考になる定番書籍

問題解決の基礎を学ぶうえで、一般に読まれている書籍をいくつか挙げます。いずれも全体像をつかむのに役立ちます。理解を深めたい人はぜひ併読してください。

- 『世界一やさしい問題解決の授業―自分で考え、行動する力が身につく』（渡辺 健介）
- 『問題解決――あらゆる課題を突破する ビジネスパーソン必須の仕事術』（高田 貴久, 岩澤 智之）
- 『ロジカル・シンキング』（照屋華子・岡田恵子）
- 『考える技術・書く技術』（バーバラ・ミント）

## 5. 次は「協働」の前提へ

問題は特定できても、関係者が動かなければ成果は出ません。だから次に大事なのは、「支配ではなく協働」という前提です。「やらされではなく、やりたい」とも言い換えられるでしょう。相手を無理に動かすのではなく、一緒に進める土台をどう作るのか。続きは[支配ではなく協働](collaboration.md)で整理します。
