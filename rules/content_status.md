# 記事ステータス運用ルール

## 目的
- 記事ごとの完成度を、読者と執筆者の両方に見える形で示す。
- 「枠組みだけある」「本文として読める」「改訂を重ねている」を区別し、次に何をすればよいか迷わないようにする。
- ステータス管理は軽く始める。細かすぎる管理で執筆を止めない。

## ページ冒頭メタ情報
- docs 配下の本文ページには、可能な範囲で冒頭に次の形式を置く。
- 読者にも見える簡易メタ情報として扱う。

```markdown
**Layer:** core
**Status:** scaffold
**Revision:** rev00
**Keywords:** ポジショニング, 目線あわせ, ラポール形成
```

## フィールド
- `Layer`
  - ページの所属領域を書く。
  - 例: `home`, `core`, `notes`, `essay`, `templates`
- `Status`
  - 記事の現在状態を書く。
  - 使用できる値は `scaffold`, `draft`, `revNN` の3系統とする。
- `Revision`
  - 改訂番号を書く。
  - 初期作成は `rev00` とする。
  - 改訂サイクルに入ったら `rev01`, `rev02` のように上げる。
- `Keywords`
  - ページの主要キーワードをカンマ区切りで書く。
  - キーワードは `rules/glossary.yml` にある語を優先する。

## Status の定義
- `scaffold`
  - 枠組みだけある状態。
  - 見出し、扱う論点、次に書く方向が分かる。
  - 本文は短くてよいが、ページの存在理由は分かる。
- `draft`
  - ある程度記事として読める状態。
  - 導入、主張、理由、落とし穴、次への導線が入っている。
  - まだ表現や例示は改訂前提。
- `revNN`
  - 公開後または利用後に改訂を重ねている状態。
  - `rev01`, `rev02` のように番号を上げる。
  - 大きな構成変更、説明の改善、用語の整理などを反映した段階。

## 運用
- 新規ページは原則として `Status: scaffold` / `Revision: rev00` で作る。
- 本文として読める状態まで育ったら `Status: draft` にする。
- 改訂サイクルに入ったら `Status: rev01` のように扱い、`Revision` も同じ番号にそろえる。
- 小さな誤字修正だけでは改訂番号を上げなくてよい。
- 説明の構成、主要キーワード、結論、導線を変えた場合は改訂番号を上げる候補にする。

## 既存の `Stability` からの移行
- 既存ページに `Stability: draft` がある場合は、新しい `Status` に置き換える。
- `Layer` は維持する。
- 移行時に内容を評価できる場合は、`scaffold` / `draft` / `revNN` のいずれかに割り当てる。
- 判断に迷う場合は `draft` ではなく `scaffold` に寄せる。
